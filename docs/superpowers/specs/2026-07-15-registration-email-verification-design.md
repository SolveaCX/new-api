# Flatkey Registration Email Verification Design

**Date:** 2026-07-15
**Status:** Approved direction, pending written-spec review
**Scope:** Registration verification email, one-click email verification, and registration-form acknowledgement

## Goal

Replace the current minimal registration email with the selected **B: Brand Welcome** design, localize it for all nine backend email locales, retain the six-character verification code, and add a secure one-click verification path that does not expose the email address or verification credential in request URLs or logs.

## Success Criteria

- Registration emails use one responsive, email-client-safe branded layout in `en`, `zh-CN`, `zh-TW`, `pt`, `es`, `fr`, `ru`, `ja`, and `vi`.
- Every email contains both a primary one-click verification button and the existing six-character code as a fallback.
- Clicking the button verifies the email in the browser without requiring the user to type the code.
- The registration form accepts either a valid code or a valid browser verification grant tied to the same email.
- An already-open registration tab recognizes the grant when it regains focus and displays an explicit verified state.
- Email security scanners cannot consume or invalidate the link by making a normal GET request.
- The flow remains correct across multiple application nodes when Redis is enabled.

## Selected Experience

### Email: B, Brand Welcome

The email uses a neutral page background with one centered, 600 px maximum-width white content surface. It presents the Flatkey wordmark first, followed by a concise welcome heading, a short verification explanation, and a solid violet primary action. Beneath the action, a clear alternative divider introduces the six-character code in a high-contrast, selectable code block. The expiry and unsolicited-request notice appear as restrained supporting text at the bottom.

The layout uses table-based HTML and inline CSS for Gmail, Outlook, Apple Mail, and common Chinese email clients. It does not use JavaScript, external fonts, gradients, decorative images, nested cards, or CSS that depends on modern browser layout engines. The button is at least 48 px high, the code remains readable on narrow screens, and the content has a 24 px mobile gutter.

Click-to-copy is not implemented inside the email because major email clients block JavaScript. The code remains selectable and copyable through the email client's normal text selection behavior.

### Browser Verification Screen

The email button opens `/sign-up/verify#token=<opaque-token>` on the configured public console origin. The verification screen uses the existing authentication layout and has three deterministic states:

- **Verifying:** spinner and localized progress text while the token is exchanged.
- **Verified:** success icon, localized confirmation, and a button back to registration.
- **Invalid or expired:** localized failure text, a button back to registration, and guidance to use the code or request another email.

After reading the fragment, the screen immediately removes it from the visible browser URL with `history.replaceState`. It never sends the fragment in a GET request.

### Registration Form

Before one-click verification, the form behaves as it does today: the user may enter the code and submit. After an email has been sent, the form checks verification status when the window regains focus. If the browser grant matches the current email, the form replaces the code requirement with a compact localized "Email verified" state. Changing the email clears that state and requires verification for the new address.

## Architecture

### Email Rendering

The HTML structure is generated by one backend email renderer so layout and escaping rules are not duplicated across nine YAML files. Backend i18n keys provide localized subject and copy fragments; the renderer receives only localized strings plus trusted runtime values such as system name, verification URL, code, and validity duration.

All interpolated text and attributes are HTML-escaped. The renderer returns a complete HTML document with inline styles. The existing SMTP sender continues to deliver it as `text/html; charset=UTF-8`.

### Credential Model

The existing six-character code remains stored under the existing email verification purpose and validity period. Sending an email also creates a cryptographically random 256-bit link token.

Two short-lived credential types are added:

- **Link token:** maps the opaque email-link token to the normalized email address.
- **Browser grant:** maps a second opaque random identifier to the normalized email address after a successful token exchange.

Redis is the authoritative store when configured. Keys contain random tokens or a SHA-256 email digest rather than a plaintext email address. Values are read and written directly through the Redis client so credential material is not exposed by debug wrappers. Local development without Redis uses the existing mutex-protected in-memory pattern. Redis write failures fail closed for the one-click path instead of reporting a usable link that another node cannot resolve.

The link token and browser grant use the same ten-minute validity window as the verification code. A per-email current-token marker invalidates older unexchanged links when a new email is sent.

### Exchange and Grant Flow

1. `GET /api/verification` validates the requested email and existing rate-limit/Turnstile rules.
2. The backend generates the code and opaque link token, stores both, renders the localized email, and sends it.
3. The email button opens the console verification route with the token in the URL fragment.
4. Frontend JavaScript extracts and removes the fragment, then sends the token in a JSON body to `POST /api/registration/email-verification/exchange`.
5. The backend validates that the link token exists, is unexpired, and is still the current token for its email.
6. The backend creates a browser grant in Redis and saves only its opaque identifier in the existing signed, HttpOnly session cookie.
7. The browser screen shows success. Re-exchanging the same valid link remains successful until expiry; the token is not consumed.
8. The registration form calls `POST /api/registration/email-verification/status` with the currently entered email when the window regains focus.
9. Registration accepts the existing code or a browser grant whose stored email exactly matches the trimmed registration email.
10. After successful account creation, the browser grant is deleted from Redis and removed from the session.

Email scanners that only request the button URL receive the SPA shell and never see the fragment. Scanners that execute JavaScript and perform the POST receive an isolated scanner-session grant, but do not consume the token, so the user's later click still succeeds.

## API Contracts

### `POST /api/registration/email-verification/exchange`

Request:

```json
{
  "token": "opaque-token-from-fragment"
}
```

Success:

```json
{
  "success": true,
  "message": "",
  "data": {
    "verified": true,
    "expires_in": 600
  }
}
```

Invalid or expired tokens return the normal API envelope with `success: false` and a localized message. The endpoint uses the anonymous body-size limit and critical rate limiter. It does not require Turnstile because possession of the high-entropy token is the proof.

### `POST /api/registration/email-verification/status`

Request:

```json
{
  "email": "user@example.com"
}
```

Response:

```json
{
  "success": true,
  "message": "",
  "data": {
    "verified": true
  }
}
```

The endpoint never reveals which token or grant exists. A missing, expired, or different-email grant returns `verified: false` rather than an authentication error.

### `POST /api/user/register`

The request shape remains backward compatible. `verification_code` stays optional at the transport layer. When email verification is enabled, the backend requires a non-empty email and accepts either a valid `verification_code` or a matching session grant.

## Localization

Backend email copy is provided for all existing email locales:

- English (`en`)
- Simplified Chinese (`zh-CN`)
- Traditional Chinese (`zh-TW`)
- Portuguese (`pt`)
- Spanish (`es`)
- French (`fr`)
- Russian (`ru`)
- Japanese (`ja`)
- Vietnamese (`vi`)

The browser verification screen and registration-form state use every locale supported by the default frontend (`en`, `zh`, `pt`, `es`, `fr`, `ru`, `ja`, `vi`). Product name, code, URL, and duration placeholders must render in every backend locale. No non-English locale may silently copy the English user-facing text.

## Error Handling

- **Email send failure:** return the existing send error and remove the newly created one-click link credentials. The code path remains consistent with current behavior.
- **Redis unavailable in a Redis-configured deployment:** do not send a one-click link that cannot work across nodes; return a retryable send failure.
- **Invalid/expired token:** show the localized browser failure state without disclosing whether the email exists.
- **Session save failure:** do not report verification success.
- **Grant/email mismatch:** treat as unverified and continue to allow the six-character code.
- **Email changed after verification:** clear frontend verified state; backend matching prevents reuse for the new address.
- **Registration failure after verification:** retain the grant until it expires so the user can correct unrelated form errors and retry.

## Security and Privacy

- The URL contains no email address, verification code, or query-string credential.
- The opaque token is placed in a fragment, which browsers omit from HTTP requests and referrer headers.
- State changes happen only through POST.
- Tokens and grants are generated with `crypto/rand` and have at least 256 bits of entropy.
- Credentials are stored with a ten-minute TTL and compared to the exact trimmed email.
- Redis keys do not include plaintext email addresses.
- Link exchange is idempotent until expiry to tolerate email scanners and double clicks.
- Responses do not disclose account existence or credential values.
- Existing email-send rate limiting, Turnstile validation, registration-domain policy, and account-existence checks remain in place.

## Test Strategy

### Backend

- Credential-store tests cover memory mode, Redis cross-instance access, expiry, current-token replacement, grant matching, deletion, and Redis failures.
- Controller tests cover valid exchange, expired/invalid exchange, session save, status matching, status mismatch, and localized errors.
- Registration tests prove that code verification remains valid, matching grants are accepted, mismatched grants are rejected, and grants are cleared only after successful registration.
- Email i18n tests prove all nine locales render every placeholder without fallback leakage.
- Renderer tests verify one complete HTML document, escaped values, responsive metadata, button URL, selectable code, inline styles, and absence of script tags.
- Router tests prove both new anonymous POST routes are registered with the intended paths.

### Frontend

- API/helper tests cover fragment parsing, fragment removal, exchange envelopes, and status envelopes.
- Verification-screen tests cover verifying, success, invalid, and missing-token states.
- Registration-form state tests cover focus refresh, matching email, changed email, code fallback, and submission without a code after a grant is confirmed.
- Frontend i18n checks ensure every new key exists and is translated in all eight interface locale files.

### Visual and Build Verification

- Render representative English, Simplified Chinese, Russian, and Japanese emails at desktop and mobile widths and inspect screenshots for overflow, missing text, broken button layout, and code readability.
- Run the targeted Go packages, frontend unit tests, frontend typecheck, i18n synchronization checks, and production frontend build.
- The current `origin/main` baseline has an unrelated failing Stripe controller assertion (`TestStripeCheckoutSessionEmbeddedModeUsesReturnURL`); this task does not modify that payment behavior, so completion evidence will separate the known baseline failure from feature-specific results.

## Deployment Scope

The change affects the Go console/API application and the embedded default console frontend. It does not alter `/v1` relay behavior, provider adapters, model routing, billing, or quota settlement. The console deployment is required. Router-node deployment is not functionally required for this feature unless the release process deploys the shared Go image to all node roles together.

## Out of Scope

- Password-reset email redesign.
- JavaScript click-to-copy inside email clients.
- Changing the verification validity duration.
- Database migrations or persistent verification records.
- Automatically creating the user account from the email click.
- Changes to registration-domain risk controls, Turnstile policy, or SMTP configuration.
