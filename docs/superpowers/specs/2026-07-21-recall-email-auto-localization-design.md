# Recall Email Automatic Localization Design

## Goal

Let an operator configure only the English recall email subject and body while the system generates and persists localized templates for every Console language. Delivery selects the recipient's snapshotted system language and falls back to English when a localized template is unavailable.

## Supported languages

The persisted template set uses the same language codes as the Console: `en`, `zh`, `es`, `fr`, `pt`, `ru`, `ja`, and `vi`.

The operator edits only `templates.en`. The seven non-English templates are generated by AI and are not directly editable in this version.

## Save flow

1. Normalize and validate the English subject and body with the existing campaign rules.
2. Compare each stage's normalized English template with the stored English template when updating an existing campaign.
3. Translation is required when the English template changed or any target-language template is missing.
4. Make one synchronous Responses API request for all stages and all seven target languages.
5. Validate the strict structured response, restore protected tokens, and normalize all generated templates.
6. Persist the complete English-plus-seven-language template map through the existing create or optimistic-update path.

The AI call runs before the database write, not inside a database transaction. Existing `config_revision` compare-and-set behavior remains the concurrency fence. Concurrent saves may duplicate an AI request, but only one database update can win.

If the English content is unchanged and all seven stored translations exist, non-template campaign edits reuse the stored localized templates and do not call AI again. If English changes, all generated translations for that stage are replaced together. Client-submitted non-English templates are never authoritative: new campaigns are translated by the server, while updates either reuse the stored server translations or replace them with a new server translation.

## AI client and configuration

Add a recall-specific translator interface to `RecallCampaignService` and a production Responses API implementation. The implementation reuses the existing monitor AI configuration:

- `operation_setting.GetMonitorAIAnalysisAPIKey()`
- `operation_setting.GetMonitorAIAnalysisBaseURL()`
- `operation_setting.GetMonitorAIAnalysisModel()`

The production client follows the existing governance-analysis client safety pattern: request context plus bounded timeout, SSRF validation, response-size limit, strict JSON Schema, sanitized error summaries, and `common.Marshal` / `common.Unmarshal` for JSON operations. Bounded retries apply only to temporary network failures, HTTP 408/429, and 5xx responses; cancellation stops immediately.

Tests inject a fake translator and never call a real AI endpoint.

## Structured output and protected content

The response contains one entry per email stage. Each entry contains exactly the seven target language keys and exactly `subject` and `body_text` for each language. Additional properties are rejected.

Before sending content to AI, URL-like text and common template-token forms such as `{{name}}` and `${token}` are replaced with stable sentinels. Every translated field must preserve the exact sentinel set. Missing, duplicated, reordered, or modified sentinels make the translation fail. The original values are restored only after validation.

Subjects must remain single-line. Existing non-empty and length validation remains authoritative for persisted templates.

## Failure behavior

When translation is required, any missing API key, timeout, non-success response, malformed structured output, missing language, stage mismatch, invalid template, or damaged protected token rejects the save. No database mutation occurs and the previous campaign version remains intact.

English fallback remains a delivery-time compatibility and resilience path for old records. It must not hide a failed translation during a new save.

## Delivery and fixed email copy

Recipient language continues to be resolved in this order and stored as `RecallRecipient.LanguageSnapshot`:

1. `User.Setting.Language`
2. `User.BrowserLang`
3. `en`

Template selection remains exact-match first, then English fallback.

`RecallEmailRenderInput` also receives the selected language. System-owned wrapper copy is translated with a static eight-language table: greeting, offer-code label, eligible-product label and product summary, expiry label, claim link, and unsubscribe link.

Claim and unsubscribe URLs remain system-generated at delivery time and never pass through AI.

## Console interaction

Remove the Template language selector. Every stage shows only its English subject and body fields. Add localized guidance explaining that saving automatically creates localized copies, delivery uses the recipient's system language, and missing localized content falls back to English.

Existing non-English templates returned by the API are kept in form state so an unrelated edit does not delete them, but only English fields are rendered and submitted as operator-controlled content.

Separately, no-group audience mode keeps the group field visible but disabled and clears `groups` immediately, preserving the canonical payload `{ "group_mode": "", "groups": [] }`.

## Compatibility and scope

- No database migration is required; templates already use a language-keyed JSON map stored in `TEXT`.
- Existing English-only records remain readable and sendable.
- Already scheduled recall messages keep their existing `TemplateSnapshot`.
- Newly scheduled messages use the newest saved templates.
- No generic background job system or manual translation override metadata is added.
- No changes are made to `main`.

## Validation

- Translator unit tests cover strict request/response behavior, configuration, retries, timeout/error handling, and protected-token validation.
- Campaign tests cover create-time generation, failure atomicity, unchanged-English reuse, missing-language repair, changed-English replacement, active-campaign version increment, and optimistic concurrency.
- Email renderer tests cover the eight static wrapper languages and English fallback.
- Frontend tests cover English-only editing, preservation of hidden localized templates, explanatory copy, and no-group normalization.
- Targeted Go and Bun tests, typecheck, lint, formatting, build, and a browser smoke test complete the release gate.
