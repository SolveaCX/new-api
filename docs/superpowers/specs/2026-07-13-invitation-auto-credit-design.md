# 邀请奖励自动到账与未解锁额度设计

- 日期：2026-07-13
- 分支：`feature/invitation-tab`
- 状态：已确认

## 背景与目标

当前邀请奖励的被邀请人部分会直接加入 API 可用额度，但邀请人部分先进入 `aff_quota`，还需要在邀请页面手动“转入余额”。产品要求取消这一步：好友完成首次成功充值后，双方奖励都应自动进入各自的 API 可用额度。

邀请页面同时按照已确认的参考图调整信息口径：

- 标题下的奖励说明继续读取当前配置。
- “如何运作”明确展示邀请人与好友分别可得多少美元 API 额度。
- 移除转入卡片、转入弹窗和新页面中的转入请求。
- 原“可转入余额”统计卡改为“未解锁额度”。
- 四张统计卡展示已赚额度、未解锁额度、已注册好友和活动状态。

## 奖励账本

好友首次成功充值触发奖励时：

1. 被邀请人奖励继续原子增加 `users.quota`。
2. 邀请人奖励改为原子增加 `users.quota`，不再增加 `users.aff_quota`。
3. 邀请人的 `users.aff_history` 继续累计实际发放金额，用于“已赚额度”。
4. `users.aff_count`、邀请次数上限、幂等事件表和首次充值触发条件保持不变。
5. 邀请人与被邀请人的缓存仍在事务提交后失效，多节点不会依赖进程内状态。

旧的 `POST /api/user/aff_transfer` 暂时保留为历史余额兼容入口，但默认邀请页面不再暴露它。这样新奖励不再需要转入，同时不会让升级前已经存在的 `aff_quota` 失去唯一的兼容处理路径。

## 未解锁额度

`GET /api/user/self/invitations` 新增 `pending_reward_usd`。服务端根据真实待充值人数和当前邀请人奖励配置计算：

- 无邀请次数上限时：`pending_count × inviter_reward_usd`。
- 有上限时：只计算未超过剩余奖励次数的待充值好友。
- 已达到上限或邀请人奖励为 0 时返回 0。

该字段代表好友完成首次成功充值后邀请人仍可能解锁的额度，不是已经可消费的余额，也不使用历史 `aff_quota`。

## 页面结构与文案

统计卡按参考图展示：

1. 已赚额度：`history_usd`，副文案“累计”。
2. 未解锁额度：`pending_reward_usd`，副文案说明好友首次充值后发放。
3. 已注册好友：分页响应 `total`，副文案使用当前邀请人奖励金额。
4. 状态：活动，副文案“跟踪中”。

“如何运作”第三步读取 `inviter_reward_usd` 与 `invitee_reward_usd`：金额相同时显示“双方各得”；金额不同时分别说明双方金额。说明文案明确奖励在首次成功充值后自动到账并可直接抵扣 API 调用费用。

FAQ 同步改为“自动加入 API 额度”，不再指导用户手动转入。

## 加载、错误与兼容

- 服务端摘要尚未返回时，配置金额相关区域继续显示骨架屏，不伪造金额。
- 摘要请求失败时，不永久显示奖励说明骨架屏。
- 分页切换沿用上一页摘要，避免统计卡闪烁。
- API 只新增字段；历史转入字段与端点暂留兼容，避免破坏旧客户端或历史余额处理。

## 国际化

所有新增或变更的可见文案同步到 `en`、`zh`、`fr`、`ru`、`ja`、`vi`、`es`、`pt` 八种语言。配置金额和次数占位符必须在所有语言中保留。

## 测试与完成标准

- 模型测试证明邀请人奖励直接增加 `quota`、不增加 `aff_quota`，并保持幂等与上限行为。
- Controller 测试覆盖无上限、剩余次数受限和已达上限的 `pending_reward_usd`。
- 前端测试证明转入 UI 不再渲染、四张统计卡口径正确、“如何运作”金额由配置驱动。
- 八种语言包含新文案并保留插值占位符。
- 邀请相关 Go/前端测试、TypeScript、目标 ESLint、Prettier、Go build 与前端生产构建通过。

## 非目标

- 不修改奖励金额、次数上限或首次充值的后台配置入口。
- 不修改 `web/classic`、website、Terraform 或 Cloudflare。
