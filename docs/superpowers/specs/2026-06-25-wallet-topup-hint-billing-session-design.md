# Wallet Top-Up Hint On BillingSession Paths

## Problem

`PR #249` added a localized recharge hint for insufficient wallet balance, but it only updated `service/pre_consume_quota.go`. The current relay runtime for paid requests goes through `controller/relay.go -> service.PreConsumeBilling -> service.NewBillingSession`, and async task submit also reaches `PreConsumeBilling` via `relay/relay_task.go`. As a result, representative paid paths such as `/v1/chat/completions`, `/v1/messages`, `/v1/responses`, `/v1/completions`, `/pg/chat/completions`, and task-submit flows still return wallet-insufficient errors without the top-up link.

The result is inconsistent user-facing behavior:

- old `PreConsumeQuota` path: may include `quota.topup_hint`
- current `BillingSession` wallet path: never includes `quota.topup_hint`

## Scope

In scope:

- wallet insufficiency errors generated by `BillingSession`
- keeping the legacy `PreConsumeQuota` path behavior consistent with the shared logic
- the `wallet_first` dual-failure edge case where wallet insufficiency is currently swallowed by subscription fallback
- regression tests for non-loopback vs known-loopback link behavior

Out of scope:

- token quota exhaustion errors
- subscription quota exhaustion errors
- changing HTTP status codes or billing semantics
- introducing new i18n keys

## Current Execution Paths

### Representative production relay requests

- `/v1/chat/completions`
- `/v1/messages`
- `/v1/responses`
- `/v1/responses/compact`
- `/v1/completions`
- `/pg/chat/completions`

The list above is representative, not exhaustive. These routes and other paid relay handlers under `controller/relay.go` eventually call `service.PreConsumeBilling(...)`, which creates a `BillingSession` and performs wallet/subscription pre-consume checks inside `service/billing_session.go`.

### Async task submit path

`relay/relay_task.go` also uses `service.PreConsumeBilling(...)` before task creation. The same missing hint can therefore surface on task-submit errors, not just synchronous relay responses.

### Legacy / compatibility path

`service.PreConsumeQuota(...)` still exists and has tests, but it is no longer the main relay path. It should remain behaviorally consistent because it still represents the older wallet pre-consume implementation and may be reused by fallback or future code.

## Options Considered

### Option 1: Patch `BillingSession` inline only

Add the same string-concatenation logic directly into `service/billing_session.go` for `userQuota <= 0`.

Pros:

- smallest code diff
- fastest to ship

Cons:

- duplicates top-up hint construction logic
- leaves `PreConsumeQuota` and `BillingSession` free to drift again
- still leaves the `userQuota-preConsumedQuota < 0` wallet error without the top-up hint

### Option 2: Extract shared wallet-insufficiency message helpers and reuse them everywhere

Move top-up hint assembly into a small shared service helper and call it from both `PreConsumeQuota` and `NewBillingSession`.

Pros:

- fixes the real production path
- keeps old and new wallet paths aligned
- covers both wallet insufficiency message shapes (`quota.user_insufficient` and `quota.pre_consume_failed`)
- small enough to keep risk contained

Cons:

- slightly larger diff than an inline patch
- touches two service files instead of one

### Option 3: Re-route wallet validation back through `PreConsumeQuota`

Make `BillingSession` delegate wallet prechecks to the old `PreConsumeQuota` implementation.

Pros:

- one canonical wallet precheck entrypoint

Cons:

- larger behavior change in a HIGH-risk shared billing surface
- risks unintended coupling with old `FinalPreConsumedQuota` semantics
- harder to reason about than a narrow message-layer fix

## Recommendation

Choose **Option 2**.

It gives us one shared wallet-insufficiency message policy without changing how quota is computed, reserved, or refunded. That keeps the fix close to the symptom while avoiding another split between the old and new billing paths.

## Proposed Design

### 1. Introduce a shared service helper for wallet-balance error messages

Create a focused helper file in `service/` that owns:

- `topUpURL()` generation
- appending `quota.topup_hint` only when `ServerAddress` is not one of the known loopback hosts we already suppress (`localhost`, `127.0.0.1`, `::1`)
- formatting the two wallet-insufficiency messages:
  - `quota.user_insufficient`
  - `quota.pre_consume_failed`

This helper should not know anything about token quota or subscription billing.

### 2. Reuse the helper in `BillingSession`

Replace the direct `common.TranslateMessage(...)` calls in `NewBillingSession` wallet prechecks with the shared helper.

This is the actual production fix for:

- representative relay routes such as `/v1/chat/completions`, `/v1/messages`, `/v1/responses`, `/v1/completions`
- Playground
- async task submit
- any other request path that uses `PreConsumeBilling`

### 3. Preserve an actionable error on `wallet_first` dual failure

Today `wallet_first` tries wallet, swallows `ErrorCodeInsufficientUserQuota`, then falls back to subscription. If subscription also fails with insufficient/no-active-subscription, the final response becomes a subscription error with no wallet hint.

The fix should make this behavior explicit:

- keep the fallback attempt, because successful subscription funding should still work
- if both wallet and subscription fail with `ErrorCodeInsufficientUserQuota`, return the original wallet error so the response keeps the actionable top-up hint
- do not invent a combined error string or new i18n key

### 4. Reuse the helper in `PreConsumeQuota`

Replace the inline message construction in `PreConsumeQuota` with the same helper.

This keeps old/new paths identical and prevents another regression where only one branch gets updated.

## Error Handling Rules

- Only wallet balance errors should receive the recharge hint.
- Token quota errors should remain unchanged because wallet recharge may not resolve them.
- Subscription insufficiency errors should remain unchanged because the recharge link is not a direct fix for that funding source.
- Loopback `ServerAddress` values (`localhost`, `127.0.0.1`, `::1`) must continue to suppress the link.
- In `wallet_first`, if wallet insufficiency and subscription insufficiency both happen in sequence, surface the original wallet error rather than the fallback subscription error.

## Testing Strategy

Add regression coverage at the service layer rather than full HTTP e2e:

1. `BillingSession` wallet insufficient error includes the top-up link for a public `ServerAddress`.
2. `BillingSession` wallet pre-consume failure error includes the top-up link for a public `ServerAddress`.
3. `BillingSession` `wallet_first` dual-failure returns the wallet-side insufficient-quota error with the top-up link.
4. `BillingSession` wallet insufficient error omits the link for loopback `ServerAddress`.
5. `PreConsumeQuota` includes the top-up hint for both `quota.user_insufficient` and `quota.pre_consume_failed`.
6. Tests assert on URL / error code behavior rather than a locale-specific message fragment.

Existing `topUpURL()` tests already cover classic/default theme path mapping and loopback filtering, so the new tests should focus on message wiring.

## Config Rollout

The allowlist is persisted in the existing `options` table through the layered config key:

- key: `topup_hint.allowed_hosts`
- value: JSON string array, for example `["console.flatkey.ai"]`

This is DB-backed config, not a GCP / Cloud Run environment variable.

### Production SQL

```sql
-- MySQL
INSERT INTO options (`key`, `value`)
VALUES ('topup_hint.allowed_hosts', '["console.flatkey.ai"]')
ON DUPLICATE KEY UPDATE `value` = VALUES(`value`);
```

```sql
-- PostgreSQL
INSERT INTO options ("key", "value")
VALUES ('topup_hint.allowed_hosts', '["console.flatkey.ai"]')
ON CONFLICT ("key") DO UPDATE
SET "value" = EXCLUDED."value";
```

```sql
-- SQLite
INSERT INTO options (`key`, `value`)
VALUES ('topup_hint.allowed_hosts', '["console.flatkey.ai"]')

SET `value` = excluded.`value`;
```

### Activation Notes

- If the value is written through the root `PUT /api/option` path, the service writes DB and publishes an immediate config-change notification.
- If the value is written by direct SQL, it does not trigger pubsub; instances pick it up on the normal options reload loop or after restart.
- Default sync frequency is `60s`, so direct SQL may take up to about one minute to become visible without a restart.

## Deployment Impact

- **Router deploy:** required
- Reason: the changed logic is in shared relay billing code used by `router.flatkey.ai/v1`
- Other deploy targets: `newapi-console` likely also picks up the change because Playground uses the same billing session path

## Success Criteria

- Wallet-insufficient errors from `BillingSession` contain the same top-up hint behavior as the legacy path.
- `wallet_first` no longer loses the actionable wallet hint when both funding sources fail.
- Non-loopback hosts render the correct theme-aware recharge URL.
- Loopback hosts still suppress the link.
- Targeted service tests cover both old and new message-construction paths.
