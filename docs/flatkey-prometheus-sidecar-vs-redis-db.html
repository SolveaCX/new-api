Flatkey Prometheus 监控方案对比

目标是监控 Cloud Run 多实例下的模型请求耗时。核心难点不是 Grafana， 而是 Prometheus 必须拿到全服务的指标，不能只抓到某一个 Cloud Run 实例。

推荐先做 Redis + DB 聚合，AWS Prometheus 只抓一个 /metrics。
保留选择 Google Managed Prometheus sidecar，用于后续需要实例级原生监控时。
避免方案 直接从 AWS 抓 Cloud Run 公网域名的 /metrics，多实例下不准确。

两个可行方案的差距

对比项 Google sidecar 方案 Redis + DB 聚合方案
数据采集方式 每个 Cloud Run 实例旁边跑 collector，抓本实例 localhost 指标。 每个实例先写本地内存，后台批量写 Redis；DB 存长期汇总。
多实例准确性 准确，天然有实例维度。 整体准确，通过 Redis 合并所有实例；实例级可用 heartbeat 补充。
能否看单实例 可以，Prometheus 有 instance / revision 等标签。 不主打，可记录实例心跳，但不做完整 per-instance histogram。
成本 按 Prometheus samples、sidecar CPU/内存、Cloud Run 实例数计费。 主要复用现有 Redis/DB；新增成本通常很小。
请求路径延迟 请求只更新本地指标，延迟很低。 请求只更新本地内存，后台批量 flush Redis，避免每次请求打 Redis。
Grafana 接入 Grafana 查 Google Managed Prometheus 或打通数据源。 继续用现有 AWS Prometheus/Grafana，只新增一个 scrape target。
故障影响 collector 或 Google Prometheus 异常会影响采集，但不影响业务请求。 Redis 异常会让新指标短时缺失；已被 Prometheus 抓走的历史仍保留。
适合阶段 适合成熟后做实例级、平台级、标准化监控。 适合现在先低成本验证模型耗时、错误率和请求量。

本次实施范围

采集维度

模型请求耗时使用 Prometheus histogram，维度控制在：

model + channel_id + status

这里故意不把 api_key 放进耗时 histogram，避免时间序列爆炸。

数据流

1. 请求结束记录 model、channel、status、latency。
2. 本地聚合只更新进程内 atomic counter。
3. 批量 flush每几秒把增量写到 Redis。
4. /metrics从 Redis 读全局聚合并输出 Prometheus 文本。
5. GrafanaPrometheus 抓取后用 PromQL 展示。

DB 的用途

现有项目已经有 perf_metrics 表，负责长期小时级/分钟级汇总， 继续用于控制台、官网模型页和历史趋势。Prometheus 实时指标走 Redis， DB 不承担每请求写入压力。

Grafana 查询示例

按模型和渠道看 P95 耗时：

histogram_quantile(
  0.95,
  sum by (le, model, channel_id) (
    rate(newapi_model_request_duration_seconds_bucket[5m])
  )
)

按模型看请求量：

sum by (model) (
  rate(newapi_model_requests_total[5m])
)

按模型和渠道看错误率：

sum by (model, channel_id) (
  rate(newapi_model_requests_total{status="error"}[5m])
)
/
sum by (model, channel_id) (
  rate(newapi_model_requests_total[5m])
)

上线配置

  • 需要配置 PROMETHEUS_METRICS_TOKEN，Prometheus 用 Authorization: Bearer <token> 抓取。
  • Redis 继续使用现有 REDIS_CONN_STRING。
  • 建议 Redis flush 间隔先用默认 5 秒，Prometheus scrape 间隔 15 或 30 秒。
  • 默认保留 channel_id 标签；如果后续时间序列太多，可以把 PROMETHEUS_METRICS_CHANNEL_LABEL_ENABLED=false 聚合到模型级。
  • PROMETHEUS_MAX_SERIES_PER_SCRAPE 默认 5000，超过时 /metrics 会标记 Redis 聚合不可用，避免单次抓取压垮 Redis。
  • 如果 Redis 不可用，/metrics 会暴露错误指标，Grafana 可以告警。

Prometheus job 示例

把下面这一段加到现有 Prometheus 的 scrape_configs 里。先验证 staging 时使用内部约定的 staging router 域名，正式上线再替换为生产 router 域名。

scrape_configs:
  - job_name: "flatkey-newapi-router"
    scheme: "https"
    metrics_path: "/metrics"
    scrape_interval: 30s
    static_configs:
      - targets: ["<staging-router-domain>:443"]
    authorization:
      type: "Bearer"
      credentials_file: "/etc/prometheus/secrets/flatkey_metrics_token"

token 不写在代码里，只放在 Cloud Run 环境变量和 Prometheus 配置里。 Prometheus 的 authorization 写法见 官方配置文档。

结论

现在先做 Redis + DB 聚合是更合适的最小验证方案：低成本、对现有 Grafana 友好、 能解决 Cloud Run 多实例整体指标不准确的问题。等后面真的需要精细到每个实例的 CPU、内存、原生实例级 histogram，再考虑 Google Managed Prometheus sidecar。
