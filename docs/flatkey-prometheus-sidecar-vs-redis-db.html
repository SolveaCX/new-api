Flatkey Google 官方 Prometheus 监控方案

目标是看清楚线上 newapi-router 多实例下的模型请求耗时、请求量和错误率。 现在方案切换为 Google 官方的 Cloud Run Managed Service for Prometheus sidecar， 不再让 AWS Prometheus 从公网随机抓某一个 Cloud Run 实例，也不再用 Redis 合并 Prometheus 指标。

最终推荐 Google 官方 Cloud Run Prometheus sidecar 抓每个实例本地 /metrics。
Prometheus 放哪 放在 Google Managed Service for Prometheus，不自己在 AWS 或 Google 起 Prometheus 服务器。
Grafana 怎么看 Grafana 接 Google Managed Prometheus / Cloud Monitoring 数据源，用 PromQL 查。

整体架构

1. 请求进来用户请求打到 Cloud Run 的 newapi-router。
2. 本地记录Go 进程把 model、channel_id、status、耗时写到进程内 counter/histogram。
3. sidecar 抓取同一个 Cloud Run 实例里的 collector 抓 localhost:3000/metrics。
4. Google 存储Managed Prometheus 存每个实例的时间序列，并带上 Cloud Run 元数据标签。
5. Grafana 展示Grafana 用 PromQL 聚合，也可以按 instance / revision 下钻。

为什么这样能解决多实例问题

Cloud Run 公网域名后面有多台实例。外部 Prometheus 去抓 https://router.flatkey.ai/metrics 时，只会命中负载均衡挑中的某一个实例，所以多实例下不准。 Google sidecar 是每个实例旁边都有一个采集器，它只抓自己旁边的 app 容器，因此每台实例都会被采集到。

和 Redis + DB 方案的差异

对比项 Google 官方 sidecar Redis + DB 聚合
Prometheus 多实例准确性 准确，每个实例各自被本地 sidecar 抓取。 需要每个实例先写 Redis，再由 /metrics 汇总，复杂度更高。
单实例定位 可以，Google 会带 Cloud Run 的 service、revision、instance 等标签。 不适合，它更像全局聚合，不天然保留每个实例的 histogram。
写入压力 业务请求只写本地内存，sidecar 定时 scrape；不增加 Redis 写入。 即使批量 flush，也会增加 Redis 写入和跨实例合并逻辑。
成本来源 sidecar 的 Cloud Run CPU/内存 + Managed Prometheus 样本/时间序列。 主要复用 Redis/DB，但 Redis 压力和实现复杂度更高。
保留 DB 历史趋势 保留。DB 继续用于控制台和长期趋势，Prometheus 只做实时监控。 保留。

代码和配置变化

  • /metrics 只输出当前 Cloud Run 实例进程内的 Prometheus counter/histogram。
  • Prometheus 指标不再通过 Redis 跨实例合并，避免多实例合并和 Redis 写入压力。
  • 来自 127.0.0.1 或 ::1 的 sidecar 本地抓取不需要 token。
  • 外部访问 /metrics 仍然要 Authorization: Bearer <token>，所以浏览器直接打开看到 401/503 是正常保护。
  • Cloud Run 主容器命名为 app，sidecar 命名为 collector。
  • RunMonitoring 配置通过 Secret Manager 挂载到 /etc/rungmp/config.yaml，抓取端口是 3000。

RunMonitoring 配置

apiVersion: monitoring.googleapis.com/v1beta
kind: RunMonitoring
metadata:
  name: newapi-router
spec:
  endpoints:
  - port: 3000
    path: /metrics
    interval: 30s
  targetLabels:
    metadata:
    - instance
    - service
    - revision

指标维度和费用控制

业务指标只放低基数标签：

model + channel_id + status

不把 api_key 放进 Prometheus label。api key 数量会持续增长，放进去会让时间序列数量快速变大。 如果后续 channel_id 也太多，可以把 PROMETHEUS_METRICS_CHANNEL_LABEL_ENABLED=false 设到 router，先聚合到模型级别。

费用主要跟时间序列数量和 scrape 频率有关。粗略理解： 

活跃时间序列 ≈ 模型数 × 渠道数 × 状态数 × 实例数 × 每个 histogram/counter 产生的序列数

当前 histogram 有 bucket、sum、count，再加 request total counter，所以 model 和 channel_id 的组合越多，费用越容易上升。默认 30 秒抓一次，先看真实 series 数，再决定是否关闭 channel 维度。

Grafana 查询示例

看每个模型 P95 耗时：

histogram_quantile(
  0.95,
  sum by (le, model) (
    rate(newapi_model_request_duration_seconds_bucket[5m])
  )
)

按模型和渠道看 P95 耗时：

histogram_quantile(
  0.95,
  sum by (le, model, channel_id) (
    rate(newapi_model_request_duration_seconds_bucket[5m])
  )
)

看某个 Cloud Run 实例是否异常：

histogram_quantile(
  0.95,
  sum by (le, instance, revision) (
    rate(newapi_model_request_duration_seconds_bucket[5m])
  )
)

看错误率：

sum by (model) (
  rate(newapi_model_requests_total{status="error"}[5m])
)
/
sum by (model) (
  rate(newapi_model_requests_total[5m])
)

上线步骤

  1. 合并本 PR，代码里会有新的 /metrics 行为和 Google sidecar Terraform 配置。
  2. 对生产 GCP Terraform 执行 plan/apply，创建 RunMonitoring secret，并给 newapi-router 加 collector sidecar。
  3. 执行一次生产 router deploy。原因是 Cloud Run revision 需要带上最新 app 镜像和 /metrics 逻辑。
  4. 在 Grafana 里添加或使用 Google Managed Prometheus / Cloud Monitoring 数据源。
  5. 导入上面的 PromQL 面板，先观察 series 数、P95、错误率，再决定是否关闭 channel_id 标签。

注意：只 apply Terraform 不够，还需要部署 router；只部署 router 也不够，还需要 Terraform 创建 sidecar。
