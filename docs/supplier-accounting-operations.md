# 供应商日结运维手册

本文档适用于[上游供应链与利润核算 V1](./supplier-supply-chain-v1.md)的发布、日常检查、自动历史 catch-up 和故障处理。

## 1. 运行形态

供应商日结运行在现有 Go 应用内：

- Router 节点在符合条件的最终成功消费日志中写入 `logs.other.supplier_accounting_v1`；
- Console/Master 节点按现有定时任务模式运行 `Asia/Shanghai` T+1 ticker；
- Router/Slave 节点不启动日结 ticker；
- 所有 Master 实例可以同时触发尝试，主库租约和 fence 决定唯一有效执行者；
- 日结从 `LOG_DB` 读日志，向主库写日报和批次状态；两者可以是同一数据库。

该运行形态不需要 Redis 锁、独立 runner、Cloud Run Job、Cloud Scheduler、Terraform 资源、专用工作流、控制面激活步骤或 Prometheus 新指标。

## 2. 发布前检查

### 2.1 数据库

确认迁移新增且仅新增设计文档列出的八张供应商表，并检查：

- 每个版本表、调整表和排除规则表的唯一约束与查询索引存在；
- 每个日结日期在 batch run 表中唯一；
- 日汇总 generation 的维度唯一约束存在；
- SQLite、MySQL 5.7.8+、PostgreSQL 9.6+ 均完成迁移验证；
- 独立日志库配置下，日志库只承担读取，供应商配置和日结状态仍写主库。

不要删除或改写历史版本、库存调整、日志快照或已发布日报来“修正”数据。库存修正使用新的库存调整记录。

### 2.2 节点角色

确认生产角色符合以下预期：

| 服务 | 角色 | 供应商核算职责 |
| --- | --- | --- |
| `newapi-router` | Slave | 处理 API 流量并写成功请求快照；不运行 ticker |
| `newapi-console` | Master | 管理接口、报表和 T+1 ticker |
| legacy `newapi` | 取决于是否仍承载流量 | 若处理 API 流量，必须具备相同快照写入能力；若为 Master，也会竞争数据库租约 |
| `newapi-web` | 网站 | 无供应商核算职责 |

即使只有一个预期 Console 实例，也必须保留数据库租约与 fence；Cloud Run 重叠修订和扩缩容会产生并发实例。

### 2.3 核算样本

发布前在隔离环境覆盖以下请求：

| 场景 | `logs.other` 预期 | 日报预期 |
| --- | --- | --- |
| 已绑定、支持、正用量的业务成功请求 | 完整快照 | 销售、采购成本、利润和库存均增加 |
| 已绑定、支持、正用量的内部排除请求 | 紧凑库存快照 | 只增加内部官方原价消耗与采购成本 |
| 请求失败 | 无供应商标记 | 不进入日报 |
| 成功但零用量/零销售额度 | 无供应商标记 | 不进入日报 |
| 最终成功渠道未绑定 | 无供应商标记 | 不进入日报 |
| 不支持的路径 | 无供应商标记 | 不进入日报 |

同时验证渠道重试：日志必须使用最终成功渠道在选择时冻结的绑定和折扣版本。

## 3. 发布顺序

1. 先完成主库的加法迁移，确认旧版本应用可忽略新增表和 `logs.other` 中的新增对象。
2. 部署 Router，使新的成功请求开始保存核算快照。若 legacy `newapi` 仍处理 API 流量，一并部署。
3. 部署 Console，使管理接口、报表和 Master ticker 使用相同协议。
4. 检查所有在服 Router 修订均已更新；混合版本期间，旧 Router 产生的日志没有标记，日结不会推断或补造这些事实。
5. 在下一个北京时间关账窗口后检查首个批次和管理报表。

该变更影响 relay 成功结算日志、供应商成本和日报，因此生产 Router 部署为必需；Console 部署也为必需。网站、Terraform 和 Cloudflare 不参与。

## 4. 每日检查

北京时间 02:00 后检查前一自然日：

1. batch run 存在，状态最终为 `completed`；
2. `published_fence_token` 大于零且对应日汇总存在；
3. `logs_scanned`、快照数和汇总行数与业务量级相符；
4. 管理端新鲜度显示最新完成日期为前一日；
5. 业务报表满足 `毛利润 = 销售额 − 采购成本`；
6. 内部报表没有销售额、毛利润或模型高维明细；
7. 合同剩余库存等于累计库存调整减去业务和内部官方原价消耗。

没有快照的成功日志可能来自未绑定、零用量、不支持路径或混合版本部署。V1 不保存这些类别的占位标记，排查时必须结合请求和部署事实，不能仅凭“无标记”确定原因。

## 5. 批次状态与处理

| 状态 | 含义 | 处理 |
| --- | --- | --- |
| `running` | 某个 owner 持有有效租约并生成候选日报 | 等待租约内完成；不要启动旁路任务 |
| `completed` | 某个 fence 已完整发布 | 报表读取该已发布 generation |
| `failed` | 扫描、解析、写入、租约或发布失败 | 修复原因；后续 ticker 会在新 fence 下重新处理未完成日期 |

### 5.1 `running` 持续过久

检查 `locked_until`、owner、fence 和 Console 日志：

- 租约仍有效时，不要手工改状态或删除汇总行；
- 租约到期后，新实例可使用数据库时间接管并递增 fence；
- 旧实例恢复后必须因 owner/fence 不匹配而停止，不能推进游标或发布；
- 如果接管失败，优先检查数据库锁、事务失败和唯一约束，不要引入 Redis 锁绕过数据库协议。

### 5.2 `failed`

按错误阶段处理：

- 日志读取失败：检查 `LOG_DB` 连接、权限和目标日期查询；
- 快照解析失败：定位异常日志，确认是否由版本不一致或数据损坏造成；
- 主库写入失败：检查磁盘、连接、锁等待、事务和约束；
- fence 丢失：通常表示另一个实例已接管，旧 owner 应正常退出；
- 发布失败：候选数据不可用于报表，保留上一已发布 generation。

不要把候选汇总直接标记为完成，也不要修改 fence 来强行发布。

### 5.3 报表缺少某一天

依次确认：

1. 该日期是否早于当天且已过 02:00 关账缓冲；
2. ticker 是否只运行在 Master/Console；
3. 是否存在更早的未完成日期；每次 ticker 最多推进一天；
4. batch run 是不存在、`running` 过期还是 `failed`；
5. `LOG_DB` 是否能读取该北京时间窗口内的消费日志；
6. 日志是否真实包含有效快照。

缺失日期不能在报表中当作零消费。

## 6. 自动历史 catch-up

自动 catch-up 用于快照写入功能上线后尚未生成汇总的日期，不用于重建上线前事实。V1 不提供手动回填、日报重跑 HTTP 接口或前端入口。

运行规则：

- 配置 `SUPPLIER_ACCOUNTING_CUTOVER_AT` 时，从该时间所在的 `Asia/Shanghai` 自然日开始寻找最早未完成日期；
- 未配置时只从 D-1 开始，不扫描更早历史；
- 日期必须早于当前 `Asia/Shanghai` 自然日；
- 每次 ticker 最多推进一个最早未完成日期，剩余日期由后续 ticker 继续处理；
- 与自动 ticker 共用同一 batch run、数据库租约、fence、分页和发布路径；
- 只读取已有 `supplier_accounting_v1` 快照，不修改 `logs.other`；
- 已完成日期不接受任意 rerun，避免在没有新的不可变事实时重写财务结果。

catch-up 完成后，按“每日检查”核对发布 fence、汇总金额和库存余额。若目标区间包含无标记日志，记录该部署事实；不要估算或人工合成请求快照。

## 7. 多节点故障场景

### 重复 ticker

多个 Master 同时触发是正常情况。只有获得主库租约的 owner 工作，其他实例应识别 busy 并退出，不得形成重复日报。

### owner 崩溃

租约到期后由新 owner 接管，使用更大的 fence 从安全边界重新生成候选数据。旧候选不能成为报表来源。

### 旧 owner 恢复

每页游标推进和最终发布都必须校验 owner、fence 和旧游标。任一不匹配即停止，禁止覆盖新 generation。

### 日志库与主库短暂不一致

日结先从日志库分页读，再在主库事务中保存候选汇总和游标。任何一侧失败都不得发布部分日报。恢复后以新租约继续未完成日期。

## 8. 回滚

应用回滚不删除八张供应商表，也不删除 `logs.other` 中已写入的快照。旧版本应忽略未知 `other` 字段。

- Router 回滚后不再产生新快照，形成明确的核算覆盖缺口；尽快恢复兼容版本；
- Console 回滚后 ticker 和新报表可能停止，但已发布日报保持不变；
- 不要通过清表、改 batch 状态、降 fence 或重写日志来回滚；
- 恢复兼容版本后，由 ticker 继续处理仍含有效快照的未完成日期。

## 9. 发布验证清单

- [ ] 三种数据库迁移和基本 CRUD 通过；
- [ ] 共享库与独立 `LOG_DB` 两种拓扑通过；
- [ ] 业务与内部快照字段隔离通过；
- [ ] 失败、零用量、未绑定和不支持路径均无标记；
- [ ] T+1 日期边界和北京时间夏令时无关语义通过；
- [ ] 两个 Master 竞争时只有一个有效 owner；
- [ ] 租约过期接管后旧 owner 无法写入或发布；
- [ ] 分页重试不漏记、不重记；
- [ ] 报表只读取已发布 fence；
- [ ] 自动 catch-up 起点和单次一天上限生效；
- [ ] 业务报表、内部库存和合同余额人工抽样一致。

验证失败时停止扩大流量，保留日志快照、batch run、候选和已发布 generation 作为排障证据。
